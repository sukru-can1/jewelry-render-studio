# Phase 5 — UI Design Contract (Outputs Gallery & Layered Passes)

> Visual and interaction contract for Phase 5. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Inheritance note (READ FIRST):** Phase 1's UI-SPEC (`01-UI-SPEC.md`, approved) is the **global design system** — token architecture (primitive → semantic → component), color (teal accent, NO purple; dark default + light), Geist Sans/Mono typography, 8pt spacing, elevation/motion, the app shell (240/56px sidebar + 56px topbar with the **Gallery** nav item, `image` glyph), and the seed component library. Phase 2 added the product card + status-pill lifecycle and the `alloycolour/diamond/stone2/stone3` **group chips** (diamond=accent, stone2=info, stone3=warning, alloycolour=neutral). Phase 3 defined the **combo shape** (`angle · metal · stoneGroup · pass`, where `metal: white|yellow|rose`, `pass: metal|stone`), the 16px metal hex swatch in a chip, and the layered-holdout passes (metal-only JPEG + one transparent PNG per selected stone group). Phase 4 built the jobs monitor, the **40×40 row thumbnail** + the **160×160 light per-job preview**, the **freshness indicator** + DB-only refresh discipline, the `scroll-area`, and — critically — **reserved the "View in gallery" link** on a completed job. **This Phase 5 spec does NOT redefine any of that.** It (1) specifies the THREE new Phase 5 surfaces (outputs gallery, output preview/lightbox, download), (2) declares the NET-NEW components those surfaces require (output/layer card with transparency checkerboard, gallery grouping + filter controls, preview lightbox, download-layer + download-set controls), and (3) reuses every inherited token, component, status-pill color, combo shape, and copy tone unchanged. Where a value is "inherited," the executor reads it from Phase 1/2/3/4, not from here.
>
> **Scale-forward note:** the gallery is the upstream of Compositing (P6). P5 only **browses, previews, and downloads** the already-separated layers (metal JPEG + per-stone transparent PNG); it does NOT stack/flatten/composite (that is P6 `COMP-01..03`). The transparency checkerboard backing and the per-layer/per-variant download grouping defined here are exactly what P6's layer-stacking preview consumes — keep them stable. P6 reserves a "Composite this variant" forward link the same way P4 reserved "View in gallery."

## DB-only + private-blob model (executor-critical, not a redefinition)

Two inherited constraints govern every interaction on these surfaces:

| Constraint | What the UI does | What the UI must NEVER do |
|------------|------------------|---------------------------|
| **DB-only reads (ORCH-02, A4)** | The gallery reads `Layer` rows (`pass`, `format`, `url`, `metadataUrl`, `isFlattened`) joined to their `Job` (`combo`, `status`, `outputPrefix`) and `Batch` (`product`, `matrix`) straight from Postgres. Layers exist only for `completed` jobs — terminal, settled. | Call RunPod `/status/:id` for any output. Re-fetch a terminal job. Poll RunPod from the gallery at all. |
| **Private-blob delivery (SEC-02)** | Every preview image and every download is fetched through the **auth-gated proxy** `GET /api/file?pathname=…` (`app/api/file/route.ts`) — it verifies the session next to `get(pathname, { access: "private" })` and streams the bytes back. Thumbnails, full-quality previews, and individual-layer downloads all resolve their `<img src>` / download href to this proxy path, derived from the stored `Layer.url` pathname. | Render a raw public Blob URL in `src`/`href`. Construct a "signed URL" — **private Blob has no time-limited-URL scheme** (RESEARCH Pitfall 5); the auth-gated proxy IS the signing mechanism. Leak the model/output pathname structure as a guessable public link. |

> **Terminology correction for downstream consumers:** the roadmap/requirements phrase "via signed URLs" is satisfied in THIS repo by the auth-gated `/api/file` proxy, not by a Blob-issued signed URL. The proxy is the access boundary. "Download the full set" (zip of a variant's layers, or all of a batch's outputs) is a **server-built archive streamed through an auth-gated route** (`/api/file` for single layers; a sibling auth-gated `…/download` route that zips by combo/variant or whole batch) — never a client-side fetch of public URLs.

### Layer → pass → format → visual treatment (the core P5 mapping; DO NOT invent)

| `Layer.pass` | `Layer.format` | Render meaning | Visual backing in gallery + preview |
|--------------|----------------|----------------|-------------------------------------|
| `metal` (alloycolour) | `jpeg` / `image/jpeg` | the opaque metal-only pass | **solid neutral card surface** (`--card`); no checkerboard (JPEG is opaque) |
| `stone` (+ `combo.stoneGroup` = diamond / stone2 / stone3) | `png` / `image/png` | a transparent-background holdout pass for one stone group | **transparency checkerboard backing** behind the image, so the operator sees alpha at a glance; the stone-group chip uses the inherited color (diamond=accent, stone2=info, stone3=warning) |

The **format badge** (`JPEG` / `PNG`, Label mono) and the **transparency-checkerboard backing on PNG only** are the two-second visual tell of "metal vs stone pass." This is the one net-new visual idiom of the phase.

## Color

**Inherited unchanged from Phase 1–4** — dark default + light, teal accent (NOT purple), 60/30/10 discipline, the full semantic + status vocabulary, and the inherited **group-chip colors** (diamond=accent, stone2=info, stone3=warning, alloycolour=neutral). P5 introduces **no new hues**. The only net-new visual surface is the **transparency checkerboard**, which is a neutral pattern (two muted greys), NOT a color in the palette.

| Role | Value (dark / light) | Phase-5 usage |
|------|----------------------|---------------|
| Dominant (60%) | `#0A0A0A` / `#FFFFFF` | gallery + lightbox-scrim base background |
| Secondary (30%) | `#141414` card · `#1F1F1F` border / `#FAFAFA` card · `#E4E4E7` border | layer cards, filter bar, grouped-section surfaces, lightbox content surface, metadata panel (slightly inset) |
| Accent (10%) | `#14B8A6` / `#0D9488` | see "Accent reserved for" |
| Destructive | `#F87171` / `#DC2626` | layer-load/download **error** state only; there is no destructive *action* in P5 (no delete in MVP) |
| Foreground | `#FAFAFA`/`#A1A1AA` muted · light `#18181B`/`#71717A` | text + secondary text + metadata values |

### Transparency checkerboard (net-new neutral pattern)

- **Dark mode:** alternating `#1A1A1A` / `#242424` 8px tiles (subtle, just enough to read alpha without competing with the artwork).
- **Light mode:** alternating `#EDEDED` / `#FAFAFA` 8px tiles.
- Applied **only behind `format = png` (stone) layers** — in the 160×160 thumbnail and the lightbox frame. JPEG (metal) layers sit on the solid `--card` surface with no checkerboard. This is the literal "transparency via a checkerboard backing" tell.

### Status / freshness colors (inherited; gallery is a terminal-only surface)

The gallery shows only **completed** jobs' layers, so the live status vocabulary mostly recedes. Where state appears:
- **partial / batch-still-rendering** banner uses `info` (rendering in progress, more outputs coming) — inherited token, NOT a new hue.
- **load/download error** uses `destructive` — inherited.
- A small **completed** `success` confirmation may appear on a freshly-finished batch's gallery header; no per-thumbnail status pill (every shown layer is, by definition, completed).

**Accent reserved for (P5 — extends Phase 1–4's list, does not broaden it):**
- The active sidebar **Gallery** nav item (inherited shell behavior).
- Keyboard focus rings (inherited 2px teal) — including focus on a thumbnail and on lightbox controls.
- The **diamond** stone-group chip (inherited from P2 — the one domain-semantic accent; stone2/stone3 stay info/warning).
- The single **primary** action where one exists: **"Download full set"** at the gallery/batch level MAY be a primary teal button (one per view max); per-layer "Download" is a **secondary/ghost** button, not teal.

**Accent stays scarce:** thumbnails, format badges, grouping tabs, and prev/next controls are neutral. The metal swatch in a combo chip renders the literal seeded metal hex (white/yellow/rose gold), not accent. The checkerboard is neutral grey, never tinted. "These are your finished renders" is communicated by the calm grid, not by painting it teal.

## Phase 5 Surfaces

Each surface names its loading / empty / partial-rendering / error / downloading states (state-completeness is the inherited bar).

### 1. Outputs gallery — reached two ways

- **Entry points:** (a) the **"View in gallery" link** Phase 4 reserved on a completed job / completed batch → opens that **batch's** outputs (`/batches/[id]/gallery` or `/gallery?batch=[id]`); (b) the inherited sidebar **Gallery** nav item → a gallery landing (most-recent completed batches, each a card linking into its outputs) — the batch-scoped view is the centerpiece; the landing is a light index.
- **Header:** Heading = **{product name} · {batch id}** (mono id) + a mono layer count (`36 layers across 12 variants`); on the right, the **"Download full set"** control (one primary teal action max for the view) and, when relevant, a quiet "back to batch" link to the P4 monitor. (The DB-only freshness model is inherited but mostly dormant here — terminal layers don't change; an optional inherited freshness chip may show "as of {time}" without polling RunPod.)
- **Filter / grouping bar:** the **gallery grouping + filter controls** — Group by **Metal · Angle · Pass · Variant** (default Metal) + filter chips (metal swatch chips / angle chips / pass chips / format JPEG·PNG) read from the live `Batch.matrix` and the present `Layer` rows.
- **Body:** **grouped gallery sections** (one per group value) of **output/layer cards** — each a 160×160 thumbnail (PNG on checkerboard, JPEG on solid), format badge, mono combo label, group chip. Whole card opens the preview lightbox.
- **States:**
  - **loading** = skeleton filter bar + 2–3 skeleton sections each with a skeleton card grid.
  - **empty (no completed outputs yet)** = centered empty-state: this batch hasn't produced any finished layers yet (heading + body pointing back to the monitor → [View batch progress]). Distinct from "no batch."
  - **partial (batch still rendering)** = the layers that ARE done are shown normally, with a calm top **`info` banner**: "{c} of {N} renders done — more layers will appear here as they finish." (does NOT poll RunPod; refreshes on navigation / inherited revalidation). Failed jobs simply have no layer rows; an optional quiet note "{f} renders failed — review them in the batch monitor → [Open monitor]".
  - **error (DB read failed)** = inline error card + Retry (the Postgres read failed — never a RunPod error surfaced as a crash).
  - **no-matches (filters)** = "No layers match these filters." + Clear filters.

### 2. Output preview / lightbox — opened from any thumbnail (OUT-03)

- Click a layer card → the **output preview lightbox**: the layer at full quality (streamed through `/api/file`), PNG on the transparency checkerboard so alpha is unmistakable, JPEG on a solid surface. Header = the combo (mono) + pass/group/format chips; a mono metadata panel (format · dimensions · size · samples · alpha-coverage % · render duration · RunPod request id). Actions: **Download layer** + a quiet "Download full set". **←/→** move prev/next through the current filtered order; **ESC** closes; focus-trapped.
- **States:** loading (skeleton frame while full-res streams) / loaded / load-error (+ Retry) / first-last (prev-next disabled at ends) / downloading (button spinner). Reduced-motion respected.

### 3. Download — individual layer OR full set (OUT-03, SEC-02)

- **Download a layer:** the per-card and in-lightbox **Download layer** action → `/api/file?pathname=…` (auth-gated proxy) with a `download` attribute + a human filename from the combo. No public URL ever.
- **Download a set:** **"Download this {metal/angle/variant}"** at the section level and **"Download full set"** at the gallery/batch level → an **auth-gated server archive route** that zips the in-scope `Layer` rows (the route re-reads from Blob server-side and streams a zip; the client never touches Blob directly). Toast feedback through the inherited `sonner`.
- **States:** idle / preparing (spinner + "Preparing download…") / started (browser download + toast) / error (+ Retry) / empty-scope (disabled + reason). A quiet "Large sets may take a moment to zip." note on the full-set action.
- **Security framing (inherited SEC-02):** all three scopes go through auth-gated routes that verify the session next to the Blob `get()`; there are no signed/public URLs; the UI never prints the raw Blob pathname.
