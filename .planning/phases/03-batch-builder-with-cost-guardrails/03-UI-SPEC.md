# Phase 3 — UI Design Contract (Batch Builder with Cost Guardrails)

> Visual and interaction contract for Phase 3. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Inheritance note (READ FIRST):** Phase 1's UI-SPEC (`01-UI-SPEC.md`, approved) is the **global design system** — token architecture (primitive → semantic → component), color (teal accent, NO purple; dark default + light), Geist Sans/Mono typography, 8pt spacing, elevation/motion, the app shell (240/56px sidebar + 56px topbar), and the seed component library. Phase 2's UI-SPEC (`02-UI-SPEC.md`, approved) adds the product workspace patterns this phase launches FROM (product card + status-pill lifecycle, the `alloycolour/diamond/stone2/stone3` group identity + group chips, the saved object→group assignment that determines which stone groups a product actually has, the hex+swatch metal editor, the number-field grid). **This Phase 3 spec does NOT redefine any of that.** It (1) specifies the FOUR new Phase 3 surfaces, (2) declares the NET-NEW components those surfaces require (multi-select toggle-chip group, stone-type picker, pass selector, the live **estimate panel**, the **cap/confirmation dialog**, the matrix summary), and (3) reuses every inherited token, component, status-pill vocabulary, and copy tone unchanged. Where a value is "inherited," the executor reads it from Phase 1/2, not from here.
>
> **Scale-forward note:** the Batch Builder is the upstream of Orchestration (P4 Jobs), Gallery (P5), Compositing (P6). A submit creates a `Batch` with `matrix` JSON + N `Job` rows (one per angle × metal × stone-assignment × pass), each with a generated recipe from `lib/enterprise-recipes.ts`. The **success state links to the batch's jobs (a Phase 4 surface that does not exist yet)** — this spec defines that link target and the batch-status pill vocabulary P4 will consume. Keep the matrix shape and combo dimensions stable.

## Domain ↔ recipe-generator binding (executor-critical, not a redefinition)

The builder is driven by the **Admin-editable DB domain** (`CameraView`, `Metal`, `StoneType`, `ObjectGroup`, `QualityPreset` — seeded per DATA-03), NOT by the literal keys hardcoded in `lib/enterprise-recipes.ts`. The operator-facing vocabulary is the DB domain; submission maps it to the recipe generator. This mapping is the contract the planner wires; the UI never shows the right-hand column.

| Operator-facing (DB domain, what the UI shows) | `enterprise-recipes.ts` input (what submit maps to) |
|------------------------------------------------|------------------------------------------------------|
| Camera views: **view1 / view2 / view3 / view4** (labels from `CameraView.label`, 4 seeded) | `angle: hero \| front \| top \| profile` (positional map; if the domain grows past 4 views the planner extends the angle set — UI must read the live `CameraView` list, not assume exactly 4) |
| Metals: **white / yellow / red** (`Metal.label` + `Metal.hex` swatch) | `metal: white \| yellow \| rose` (`red → rose`) |
| Stone groups present on THIS product: subset of **diamond / stone2 / stone3** (from saved `ObjectGroupAssignment`) | `stoneGroup` + `stoneMaterials: { diamond, stone2, stone3 }` |
| Stone type per group (from `StoneType` catalog) | `EnterpriseStoneMaterial: diamond \| sapphire \| emerald \| ruby` (catalog key maps to a supported material; planner validates) |
| Passes: **metal-only (alloycolour)** + one transparent pass **per selected stone group** | `pass: metal` (one) + `pass: stone` with `stoneGroup` (one per selected group). `full` is NOT exposed in P3 — layered holdout passes only. |
| Quality preset: **preview / medium / high / ultra** (`QualityPreset.samples` + `width`/`height`) | `samples` + `resolution` |

**Job-count formula (the number the estimate panel renders):**
`jobCount = |angles| × |metals| × |passes|`, where `|passes| = 1 (metal-only) + (# selected stone-group passes)`. Stone-**type** choice per group selects the material per pass; it does not multiply job count (one stone type per group per batch). This matches BATCH-07's "one job per (angle × metal × stone-assignment × pass)".

## Spacing Scale

**Inherited unchanged from Phase 1/2** (8-point scale: xs 4 / sm 8 / md 16 / lg 24 / xl 32 / 2xl 48 / 3xl 64; Tailwind `1/2/4/6/8/12/16`). Layout constants (240/56 sidebar, 56 topbar, 1280 content max-width, 24px gutters, 8px card radius, 6px pill radius) inherited.

| Token | Value | Phase-3 application |
|-------|-------|---------------------|
| xs | 4px | chip inner gaps, swatch-to-label gap, count-segment spacing in the matrix summary |
| sm | 8px | toggle-chip vertical padding, estimate-row key/value gaps, stone-type grid cell padding |
| md | 16px | selector-card padding, gap between selector sections, dialog body rhythm |
| lg | 24px | builder section spacing, gap between the selector column and the estimate panel |
| xl | 32px | page-header → builder gap, estimate panel internal top padding around the big number |
| 2xl | 48px | major break between matrix selectors and the submit footer |
| 3xl | 64px | empty-state vertical padding (no-assignment guard) |

**Phase-3 layout (NET-NEW pattern — a two-column builder):**
- **Two-pane builder at ≥1024px:** left **selector column** (flex, ~ 60–66% width) holds the matrix selectors (angles → metals → stone types → passes → quality); right **estimate rail** (~ 320–360px fixed) is **`position: sticky; top: 56px+24px`** so the live count/cost stays visible while the operator scrolls the selectors. Below 1024px the estimate panel **docks to the bottom as a sticky bar** (full-width, condensed) so the guardrail is never scrolled off-screen — the cost readout must always be on screen (the whole point of the guardrail).
- Toggle chip: min 44px hit area (inherited rule), 6px radius (pill-family), 8px vertical / 12px horizontal padding.
- Metal swatch in a chip: 16px rounded square (inherited Phase 1 swatch size; Phase 2's 24px editor swatch is the editor exception, not reused here).

**Exceptions:** inherited 44px min hit-area applies to every toggle chip, stone-type option, and pass toggle. 1px hairlines exempt. The estimate panel's **big number** type size is a declared typographic exception (see Typography).

## Typography

**Inherited unchanged from Phase 1/2** — four roles, two weights (400 / 600), tabular-nums on all numeric fields. Geist Mono on all guardrail numerics.

| Role | Size | Weight | Line Height | Phase-3 usage |
|------|------|--------|-------------|---------------|
| Body | 14px | 400 | 1.5 | selector labels' helper text, chip labels, dialog body copy, consequence copy |
| Label | 12px | 600 | 1.4 | section captions ("ANGLES", "METALS", "STONE TYPES", "PASSES", "QUALITY"), chip-group headers, estimate-row keys (uppercase 0.04em), status-pill text |
| Heading | 20px | 600 | 1.3 | page title ("Build batch"), product-context subhead, dialog title ("Create N jobs?") |
| Display | 28px | 600 | 1.2 | **(see exception)** not used for page hero in P3 |

**NET-NEW typographic exception — the estimate "big number":** the live **job count** in the estimate panel renders at **Geist Mono 600, 40px, line-height 1.1, tabular-nums** — one step above Display, deliberately oversized so a large number reads as physically big/alarming. This is the single largest type in the app and is reserved exclusively for the estimate panel's primary job-count readout. Est. GPU minutes and est. cost render directly beneath at **Heading-size mono (20px / 600)**. All other estimate values (per-dimension counts) are Body-mono (14px / 400). The big number is the only place this 40px size appears.

- **Mono (Geist Mono):** every number in P3 — job count (40px), est. minutes + cost (20px), per-dimension counts (14px: "4 angles × 3 metals × 4 passes"), sample counts, resolution (`1920×1920`), the hard-cap and threshold constants in copy.

### Estimate-panel escalation — maps to inherited STATUS tokens (do NOT invent colors)

The estimate panel's big number changes its **accent ring/treatment by zone**, using the inherited status vocabulary so escalation reads instantly and color-blind-safely (the count number itself stays high-contrast `--foreground`; the *frame/indicator* carries the status hue):

| Zone | Condition | Inherited token | Visual treatment of the estimate panel |
|------|-----------|-----------------|------------------------------------------|
| **idle / empty** | no selection complete yet | `neutral` / idle | muted panel, number shows `0`, submit disabled |
| **under cap (safe)** | `0 < jobCount ≤ SOFT_THRESHOLD` | `success` | calm — thin neutral/teal frame, no warning; submit enabled |
| **over soft threshold** | `SOFT_THRESHOLD < jobCount ≤ HARD_CAP` | `warning` (amber) | amber left-border + amber `triangle-alert` glyph beside the number + inline notice; submit enabled but routes through the confirmation dialog |
| **over hard cap (blocked)** | `jobCount > HARD_CAP` | `destructive` (red) | red frame + red `triangle-alert` + the big number rendered in `--destructive`; **submit disabled**; blocking `alert` with guidance to narrow the selection |
| **submitting** | submit in flight | `info` / running | panel shows inline progress; selectors + submit disabled |

**Threshold constants (declared here, exact values are planner/REQUIREMENTS-driven; UI reads them from a single config, never hardcoded in the component):** `SOFT_THRESHOLD` (confirmation required above this) and `HARD_CAP` (submit blocked above this) per BATCH-06. Recommended defaults to plan against: **SOFT_THRESHOLD = 48 jobs**, **HARD_CAP = 200 jobs** (48 ≈ the full 4×3×4 grid as the "are you sure" line; 200 as the absolute ceiling). The numbers are surfaced verbatim in copy ("over the 200-job limit").

**Accent reserved for (P3 — extends Phase 1/2's list, does not broaden it):**
- The single primary button per view — **"Create batch"** (submit) — and the confirmation dialog's primary — **"Create N jobs"**.
- A **selected** toggle chip (angle/metal/pass): teal 1px ring + faint teal tint + teal check — the only place selection accent appears; unselected chips are neutral surface + border.
- The active sidebar **Batches** item (inherited shell behavior).
- Keyboard focus rings (inherited 2px teal).

**Accent stays scarce:** the estimate big number is NEVER teal — under-cap it's plain `--foreground` with a success cue; over-threshold/cap it shifts to amber/red status, never accent. Metal chips render the literal seeded metal hex swatch (white/yellow/red gold), not accent. "Cost is fine" is communicated by the *absence* of a warning, not by painting things teal.

## Phase 3 Surfaces

Each surface names its loading / empty / invalid / under-cap / over-threshold / over-cap / submitting / success / error states (state-completeness is the inherited bar).

### 1. Batch builder — `/products/[id]/batches/new` (launched from a product detail "Build batch" primary)

- **Entry:** the **"Build batch"** primary button on product detail (Phase 2 surface) is enabled ONLY when the product status is **ready** (has a saved object→group assignment). For non-ready products the button is present but disabled with a tooltip pointing back to Groups.
- **Header:** Heading **"Build batch"** + a product-context subhead (product name + thumbnail + status pill, inherited) so the operator knows which product they're fanning out.
- **Selector column (left / top):** stacked sections, each a flat `Card` with a Label caption:
  1. **Angles** — Multi-select toggle-chip group over the live `CameraView` list (view1–4), each chip = label + mono az/el hint. Default: **all views pre-selected** is NOT used — default to **view1 (hero) only** to keep the starting estimate small (guardrail-first). At least one required.
  2. **Metals** — Multi-select toggle-chip group over the 3 metals, each chip = label + 16px hex swatch. Default: **white only** (smallest safe start). At least one required.
  3. **Stone types** — Stone-type picker, one row per stone group the product actually has (diamond/stone2/stone3 from the saved assignment). Each defaults to a sensible catalog stone type for that group. Hidden entirely if the product has no stone groups (metal-only product → only the metal pass).
  4. **Passes** — Pass selector: metal-only (default ON) + one toggle per present stone group (default ON). At least one required.
  5. **Quality** — `select` over `QualityPreset` (preview / medium / high / ultra). **Default: preview** (BATCH-06 cost guardrail) with a quiet note "Preview is fast and cheap — switch up only when you need final quality." Shows mono samples + resolution.
- **Estimate rail (right / docked):** the Live estimate panel — sticky, always visible, live-updating.
- **Footer:** Matrix summary read-back + **"Create batch"** primary (right) + "Cancel" ghost (left). Submit behavior depends on zone (see states).
- **States:**
  - **no-assignment guard (empty):** if reached for a product without a saved group assignment → centered empty-state, no selectors. Copy: "This product doesn't have its parts assigned to groups yet. Assign them first, then build a batch." → [Go to groups] (links to the product's Groups tab). This is the inherited empty-state pattern.
  - **loading:** domain (camera views / metals / stone types / quality) + product assignment loading → skeleton selector cards + skeleton estimate panel.
  - **invalid selection:** any required dimension empty (zero angles / zero metals / zero passes) → that section shows an inline group error, the matrix summary greys the missing dimension, the estimate big number shows `—` (not a misleading 0 cost), and submit is disabled. Copy: "Select at least one {angle / metal / pass} to continue."
  - **under cap (safe):** `0 < jobCount ≤ SOFT_THRESHOLD` → success-framed estimate, submit enabled, one-click create (no confirm).
  - **over soft threshold:** `SOFT_THRESHOLD < jobCount ≤ HARD_CAP` → amber estimate + inline notice; **"Create batch"** stays enabled but opens the **confirmation dialog** before submitting.
  - **over hard cap (blocked):** `jobCount > HARD_CAP` → red estimate + blocking `alert`; submit **disabled**; guidance copy names the levers.
  - **submitting:** on confirm/submit, selectors + submit disabled, estimate panel shows inline progress (info), button shows loading spinner + "Creating batch…".
  - **success:** batch + jobs created transactionally → success toast ("Batch created — {N} jobs queued.") and **redirect to the batch's jobs** (`/batches/[batchId]` — a Phase 4 surface; until P4 exists the link target is reserved and the toast still confirms creation). The batch's initial status pill = **queued** (inherited `warning`/queued vocabulary; P4 owns progression).
  - **error:** transactional create failed (all-or-none, nothing partially created) → destructive banner: "Couldn't create the batch. Nothing was submitted — try again." → [Try again]. The estimate/selection is preserved so the operator doesn't re-enter it.

### 2. Live estimate panel (the cost guardrail centerpiece) — component within surface 1

- Always on screen (sticky rail ≥1024px / docked bottom bar <1024px). Renders the **40px mono big number** = current job count, the formula line, est. GPU minutes, est. cost, and escalates frame color by zone (neutral → success → amber → red) per the escalation table. The number animates its value on change (count-up is optional and must honor `prefers-reduced-motion` → instant). The escalation is the visual mechanism that makes "you're about to launch 312 jobs" feel different from "you're about to launch 3 jobs."
- **Cost/time basis tooltip** (inherited `tooltip`): "Estimated from {samples} samples at {W}×{H} across {N} jobs. Actual GPU time varies." — never presented as a guaranteed price.

### 3. Hard cap + confirmation — components within surface 1

- **Soft threshold → confirmation dialog:** see Cap/confirmation dialog component. Triggered on submit only in the amber zone. Consequence copy is unambiguous (count + ~time + ~cost). Primary is **"Create {N} jobs"** (teal — confirmed, not destructive). Cancel returns to the builder with selection intact.
- **Hard cap → blocking state:** see Over-cap blocking alert. Submit disabled; the only path forward is narrowing the selection; the estimate panel and alert both name the exact limit (mono).

### 4. Submit + result — within surface 1

- **Submit** creates the `Batch` (`matrix` JSON snapshot of the selection + `jobCount`) and N `Job` rows transactionally (BATCH-07, all-or-none). Each job carries its `combo` (angle/metal/stoneGroup/pass) and a generated `recipe` from `lib/enterprise-recipes.ts` via the domain↔recipe binding above.
- **Success → link to the batch's jobs** (Phase 4 surface). **Failure → clear, reassuring error** that confirms nothing was created (transactional safety made visible).
