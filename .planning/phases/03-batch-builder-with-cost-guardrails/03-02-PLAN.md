<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> createBatch Server Action | Untrusted selection + productId; the authoritative AuthZ + validation + cap boundary |
| validated selection -> RunPod cost (Phase 4) | The job count created here directly drives GPU spend downstream; the server cap is the gate |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Denial of Service (cost) | createBatch job-count | mitigate | Recompute `countJobs` server-side from the validated selection; reject `> HARD_CAP` before any write (BATCH-06; client number never trusted) |
| T-03-05 | Elevation / Information disclosure | productId IDOR | mitigate | `product.findUnique` + reject missing or `status !== "ready"`; never act on a client-supplied product blindly (V4) |
| T-03-06 | Tampering | client-supplied recipe JSON | mitigate | The action NEVER accepts a recipe from the client; recipes are generated server-side by `buildEnterpriseRecipe` from validated keys only (RESEARCH §threat patterns) |
| T-03-07 | Tampering / Integrity | unsupported StoneType -> bad material_map | mitigate | `resolveStoneMaterial` null -> reject before write (Pitfall 3) |
| T-03-08 | Integrity (replay) | double-submit duplicate batch | mitigate | Client disables submit in-flight (UI-SPEC submitting state, wired in 03-03); action is otherwise idempotent-safe per request — no partial batch on failure (all-or-none) |
| T-03-09 | Tampering (partial write) | non-atomic fan-out | mitigate | Single `prisma.$transaction`; a throw mid-tx rolls back the Batch and all Jobs (BATCH-07) |
| T-03-SC | Tampering | npm installs | accept | No new npm runtime deps (RESEARCH audit: none) |
</threat_model>
