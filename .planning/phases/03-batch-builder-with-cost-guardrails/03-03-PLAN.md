<objective>
Build the operator-facing Batch Builder surface: an RSC page that loads the live domain
and the product's saved assignment (with a no-assignment guard), a client builder with
multi-select angle/metal/pass chips, a per-present-group stone-type picker, a quality
select defaulting to preview, and the always-visible live estimate panel with zone
escalation, soft-threshold confirm dialog, and hard-cap blocking. Submit calls the
`createBatch` action from Wave 1. Add the "Build batch" launch button to product detail.

Purpose: this is the vertical slice that makes the guardrail real and visible — the whole
point of the phase is that the operator SEES the cost before a click fans out hundreds of
GPU jobs. The server cap (Wave 1) is the backstop; this surface is the prevention.

Output: two new shadcn components, the builder page + client + estimate panel, the launch
button, the wiring into product detail, and a builder integration test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser builder -> createBatch action | Client selection is advisory; the server (03-02) re-validates and re-caps — this surface never holds authority |
| Admin-editable domain -> builder UI | Domain rows render as operator-facing labels; recipe keys are never shown (UI-SPEC binding contract) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-10 | Denial of Service (cost) | client cap UI | mitigate | Client gate (disable >HARD, confirm >SOFT) is UX; the authoritative cap is server-side (03-02 T-03-04). UI uses the SAME `BATCH_LIMITS` config — no divergent constant |
| T-03-11 | Tampering | stone-type picker | mitigate | Picker is restricted to `isSupportedStoneType` keys; server re-validates (03-02 T-03-07) so a crafted request still fails closed |
| T-03-12 | Elevation | builder page reachable by URL for any product | mitigate | RSC `requireSession()` first line; the server action re-checks IDOR + ready-status regardless of what the page rendered |
| T-03-13 | Integrity (replay) | double-click submit | mitigate | Submitting state disables selectors + submit button while the action is in flight (UI-SPEC submitting state) |
| T-03-SC | Tampering | shadcn add toggle-group/alert | mitigate | Official shadcn/ui registry only (UI-SPEC §Registry Safety); first-party component copies, no third-party registry, no npm runtime dep added |
