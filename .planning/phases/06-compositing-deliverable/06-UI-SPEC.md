# Phase 6 тАФ UI Design Contract (Compositing & Deliverable)

> Visual and interaction contract for Phase 6. Generated by gsd-ui-researcher.
>
> **Inheritance note (READ FIRST):** Phases 1тАУ5's UI-SPECs are the locked design system and are **not redefined here**. ## DB-only + private-blob model (inherited, executor-critical)

Same two constraints that governed Phase 5 govern every Phase 6 interaction тАФ restated, not redefined:

| Constraint | What the UI does | What the UI must NEVER do |
|------------|------------------|---------------------------|
| **DB-only reads** | The compositor reads the same `Layer` rows (`pass`, `format`, `url`, `metadataUrl`, `isFlattened`) joined to their `Job.combo` straight from Postgres тАФ exactly what `loadBatchGallery()` already returns. Layers exist only for `completed` jobs. The flattened deliverable, once produced, is itself a `Layer` row with `isFlattened = true`. | Call RunPod for any preview/composite. Poll the GPU dispatch client from this surface at all. |
| **Private-blob delivery** | Every preview `<img src>` and every download href resolves through the auth-gated proxy via `privateUrl(layer.url)` тЖТ `GET /api/file?pathname=тАж`. The whole-batch and per-variant deliverable downloads go through the **auth-gated server zip/attachment route** that re-reads from Blob server-side. | Render a raw public Blob URL. Construct a "signed URL" for delivery (private Blob has no time-limited-URL delivery scheme тАФ the proxy IS the boundary). Fetch Blob from the client. |

### The compositing variant (the core Phase 6 idiom тАФ DO NOT invent)

A **compositing variant** = one **(camera angle ├Ч metal)** pair. Its layers, drawn from that variant's `completed` jobs:

| z-index | Layer | `pass` / `format` | Stacking treatment |
|---------|-------|-------------------|--------------------|
| **bottom (z=0)** | the opaque metal pass (alloycolour) | `metal` / `jpeg` | the base; on the solid composite surface, **always visible** (the floor of the stack) |
| **over (z=1+)** | each transparent stone-group PNG (diamond / stone2 / stone3) | `stone` / `png` | absolutely-positioned `<img>` layered over the metal, `object-contain`, one z per group |

The in-browser preview (COMP-01) stacks these as absolutely-positioned `<img>` elements with ascending z-index inside one frame; per-layer toggles control each layer's visibility/opacity. The frame uses the inherited **`gallery-checkerboard`** backing so any uncovered alpha reads at a glance, exactly as in the Phase 5 lightbox. This stack is **client-only composition for preview** тАФ the catalog-ready flatten (COMP-02) happens server-side.

## Color

**Inherited unchanged from Phases 1тАУ5** тАФ dark default + light, teal accent (NOT purple), 60/30/10 discipline, full semantic + status vocabulary, and the inherited **group-chip colors** (diamond=accent, stone2=info, stone3=warning, alloycolour=neutral). Phase 6 introduces **no new hues**. The composite preview frame reuses the existing neutral **`gallery-checkerboard`** pattern.

| Role | Value (dark / light) | Phase-6 usage |
|------|----------------------|---------------|
| Dominant (60%) | `#0A0A0A` / `#FFFFFF` | compositing surface base background |
| Secondary (30%) | `#141414` card ┬╖ `#1F1F1F` border / `#FAFAFA` card ┬╖ `#E4E4E7` border | variant compositor cards, the toggle panel surface, the metadata strip |
| Accent (10%) | `#14B8A6` / `#0D9488` | see "Accent reserved for" |
| Warning | `#FBBF24` / `#B45309` (with `warning-foreground`) | **the COMP-02 WARN banner only** тАФ mismatched dimensions / missing metal layer / empty alpha. This is the one semantic color this phase leans on. |
| Destructive | `#F87171` / `#DC2626` | flatten/download **error** state only (the operation failed) тАФ distinct from WARN (which is a soft caution that still lets the operator proceed). |
| Info | `#38BDF8` / `#0369A1` | the "batch still rendering тАФ not all variants composable yet" partial banner (inherited idiom). |
| Foreground | `#FAFAFA`/`#A1A1AA` muted ┬╖ light `#18181B`/`#71717A` | text + secondary text + mono metadata |

**Accent reserved for (Phase 6 тАФ extends Phases 1тАУ5's list, does not broaden it):**
- Keyboard focus rings (inherited 2px teal) тАФ including focus on each layer toggle and on the action buttons.
- The **diamond** stone-group chip (inherited domain-semantic accent; stone2/stone3 stay info/warning).
- The single **primary** action per view: **"Download all deliverables"** at the batch level MAY be a primary teal button (one per view max). The per-variant **"Flatten & download"** is a **secondary** button, not teal.

**Accent stays scarce:** the preview frame, toggles (off = neutral, on = neutral filled), variant card chrome, and the WARN/partial banners are neutral or their own semantic token тАФ never painted teal. The metal swatch (if shown in a variant chip) renders the literal seeded metal hex, not accent.

**WARN vs ERROR vs INFO (the one distinction the executor must keep straight):**
- **WARN (`warning`)** = COMP-02: layers exist but are *suspect* (dimension mismatch, missing metal base, empty/near-empty alpha). The flatten is **not blocked silently** тАФ the UI shows the warning and lets the operator proceed deliberately or fix upstream. Soft, recoverable.
- **ERROR (`destructive`)** = the flatten/download *operation itself failed* (server error, blob read failure). Hard, retry.
- **INFO** = the batch isn't fully rendered, so some variants aren't composable yet. Neutral progress.

## Layout & Placement

**Decision: a `compositing/` route segment alongside `gallery/`** тАФ `app/(app)/batches/[id]/compositing/page.tsx` тАФ NOT a client tab inside the gallery page. **One-line justification:** compositing is a heavier, stateful, write-capable surface (server flatten + per-variant deliverable state) that warrants its own DB-only Server Component entry, IDOR-loaded by `params.id` exactly like `gallery/page.tsx` тАФ mirroring the existing route shape keeps the auth/load/`force-dynamic` discipline identical and avoids bloating the gallery client bundle.

- **Navigation between the two:** the batch-detail area gets a lightweight **segment switcher** (the inherited `toggle-group` or simple links) at the top of both the gallery and compositing pages тАФ **Monitor ┬╖ Gallery ┬╖ Compositing** тАФ so an operator moves between the Phase 4 monitor, the Phase 5 gallery, and this surface. Phase 5 reserved a "Composite this variant" forward link; that link targets `тАж/compositing` (optionally `?variant=view1-white`).
- **Page header:** Heading = **{product name} ┬╖ {batch id} тАФ Compositing** (mono id) + a mono summary (`{v} variants ┬╖ {d} flattened`); on the right, the **"Download all deliverables"** primary action (one per view).
- **Body:** the **partial banner** (info) when the batch is still rendering, then a list of **variant compositor cards** тАФ one `LayerCompositor` per (angle ├Ч metal) variant, grouped by metal (reusing the `groupLayers()` pattern; note Phase 6's variant bucket is **angle ├Ч metal**, which the existing `metal` group mode already keys by `metalKey:angleKey` тАФ reuse that mode directly rather than the `variant` mode).
- **Reuse note:** the existing `PreviewLightbox` (Phase 5) opens a single raw layer; the compositor's *composed* preview is a NET-NEW frame (stacked `<img>`), so it is its own component, but it borrows the same checkerboard, `privateUrl`, chips, and combo-label helpers.

## Phase 6 Surfaces & States

The required five states (UI-02 carryover) per surface, with concrete copy.

### 1. Compositing page (per-batch)

- **Entry:** the **Monitor ┬╖ Gallery ┬╖ Compositing** segment switcher, or Phase 5's reserved "Composite this variant" link.
- **States:**
  - **loading** = skeleton header + 2тАУ3 skeleton variant compositor cards (skeleton frame + skeleton toggle rows).
  - **empty (no completed layers yet)** = centered empty-state: this batch has no composable layers yet, pointing back to the monitor.
  - **error (DB read failed)** = inline error card + Retry (the Postgres read failed тАФ never a GPU error surfaced as a crash). Reuses the exact `gallery/page.tsx` "Couldn't load" pattern.
  - **in-progress (batch still rendering)** = composable variants shown normally; a calm top **`info` banner** that more variants will become composable as renders finish (does NOT poll RunPod).
  - **WARN (COMP-02)** = surfaced per-variant inside each `LayerCompositor` (see component) тАФ the page itself is not the warning owner.

### 2. LayerCompositor (per variant) тАФ COMP-01

- Stacked `<img>` preview on the checkerboard; per-layer `eye`/`eye-off` toggles; metal base default-on (floor), all stones default-on; toggling sets layer opacity 1тЖФ0.
- **States:** loading (skeleton frame + toggle rows) / composed (default-on stack) / per-layer toggled-off / per-layer image-load error (+ Retry, rest still composes) / WARN (banner above frame, best-effort stack still shown).

### 3. Flatten & download (per variant) тАФ COMP-02 + COMP-03

- Server-side flatten тЖТ auth-gated deliverable download. Empty/mismatched layers WARN (not silent).
- **States:** idle / flattening (spinner) / ready (+ "Deliverable ready" badge + download starts) / warn-present (enabled + inline caution) / error (+ Retry) / empty-scope (disabled, "Flatten some variants first." / "No metal base for this variant.").

### 4. Download all deliverables (per batch) тАФ COMP-03

- Auth-gated zip of the batch's flattened deliverables (reuses `download/route.ts`).
- **States:** idle / preparing (spinner) / started (browser download + toast) / error (+ Retry) / empty-scope (disabled until something is flattened).
