<task type="checkpoint:decision" gate="blocking">
  <name>Task 1: Decide deliverable persistence (resolve the Layer.jobId @unique + FK conflict)</name>
  <decision>How to persist the flattened deliverable, given the live schema constraint.</decision>
  <context>
    The brief proposed persisting deliverables as a `Layer` row with `isFlattened=true`
    and a synthetic `jobId="deliverable:<batchId>:<angle>:<metal>"`, and asked us to
    confirm this against the schema. CONFIRMED INFEASIBLE AS LITERALLY STATED:
    `Layer.jobId` is BOTH `@unique` AND a REQUIRED foreign key to `Job.id`
    (`schema.prisma:176-177`). A synthetic jobId string has no matching `Job` row, so the
    FK insert is rejected. Reusing the variant's real metal-pass `Job.id` collides with
    that job's existing source `Layer` (`@unique`). So "isFlattened Layer + synthetic
    jobId, no migration" cannot both hold. Two clean resolutions:
  </context>
  <options>
    <option id="blob-only">
      <title>Option B — deliverable as a private blob only (NO migration, recommended)</title>
      <pros>Honors "NO Prisma migration" literally. No FK/unique conflict. Idempotent via `putPrivate(allowOverwrite:true)`. Deliverables live at a deterministic pathname `renders/&lt;batchId&gt;/deliverables/&lt;angle&gt;_&lt;metal&gt;.png`; existence/listing via `head(pathname)` or `list({prefix})`. The gallery/zip find deliverables by the `deliverables/` prefix — no DB row needed. Smallest possible change; MVP-tight.</pros>
      <cons>`isFlattened` Layer rows are never written, so `loadBatchGallery().layers[].isFlattened` stays all-false. The compositing page must list deliverable blobs by prefix instead of querying `isFlattened`. The UI-SPEC's "deliverable is itself a Layer row with isFlattened=true" line becomes "deliverable is a blob under deliverables/" (documented swap).</cons>
    </option>
    <option id="migration">
      <title>Option A — small migration: nullable jobId + deliverable identity</title>
      <pros>First-class deliverable rows; `isFlattened=true` queryable in SQL; matches the original RESEARCH/UI-SPEC wording exactly.</pros>
      <cons>Requires a real Prisma migration (make `jobId` nullable, add `batchId/angleKey/metalKey`, add `@@unique([batchId,angleKey,metalKey,isFlattened])`, keep source-layer uniqueness via a filtered/partial index) against the live Railway DB — the brief explicitly said NO migration. Heavier; not MVP.</cons>
    </option>
  </options>
  <resume-signal>Select: blob-only (default, no migration) or migration. The plan tasks below assume blob-only; if migration is chosen the executor adds the schema change + a BLOCKING `prisma migrate` step and persists an isFlattened Layer instead.</resume-signal>
</task>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → flatten route | Untrusted `angle`/`metal`/`force` query crosses into the route; layer set must be derived from DB, not caller pathnames |
| private Blob → sharp | Worker render bytes (trusted origin) decoded by libvips for compositing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Spoofing | unauthed flatten | mitigate | `requireSession()` first line, fail-closed 401 (comp-flatten-auth test) |
| T-06-02 | Elevation/IDOR | flatten another project's batch | mitigate | batch loaded by params.id; variant layers derived from that batch's DB rows only, never caller pathnames |
| T-06-03 | Tampering | path traversal via angle/metal in deliverable pathname | mitigate | `deliverablePathname` sanitizes the `<angle>_<metal>` stem (strip CR/LF/quotes/separators) before put |
| T-06-04 | Information Disclosure | composite an arbitrary guessed blob | mitigate | layers come from groupVariantsForCompositing over DB rows; get() only the DB-sourced pathnames; private read only |
| T-06-05 | DoS | decompression bomb / oversized layer | accept/mitigate | inputs are our own worker renders (trusted); one variant per request + 60s budget; libvips default pixel limits |
</threat_model>
