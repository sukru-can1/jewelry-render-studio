# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope note:** Phase 1 is the FIRST UI-bearing phase of a UI-heavy product. This contract is therefore the **global, foundational design system** (token architecture, color, typography, spacing, elevation, motion, the seed component library, and the app shell) that EVERY later phase (Product Workspace, Batch Builder, Job Monitor, Gallery, Compositing) reuses — plus the concrete Phase 1 surfaces (login, app shell, user management, domain settings). Later phases extend this contract; they do not redefine it. Phase 7 ("UI Design System & Workflow Polish") is the polish/consistency pass over the system established here, not a from-scratch system.
>
> Design intelligence: built with `ui-ux-pro-max` design-system thinking (primitive → semantic → component tokens). Design language: **Vercel** (crisp neutral surfaces, sharp 2px focus rings, monospace accents for technical values) + **Notion** (calm hierarchy, generous whitespace, sidebar-driven nav) + **RunPod** (operational dashboard feel: status pills, dense tables, compute affordances). **No purple** anywhere as brand/primary/accent.

### shadcn Setup (planner action, not executed here)

This is a fresh app (no `components.json`, no `tailwind.config`, no `globals.css` detected). The planner should:

1. Add Tailwind CSS v4 + `next/font` (Geist), then `npx shadcn@latest init` with: style `new-york`, base color `neutral`, CSS variables `yes`, RSC `yes`, `@/components` + `@/lib/utils` aliases.
2. Seed the **CSS variable token layer** exactly as defined in "Token Architecture" below into `app/globals.css` (`:root` = light, `.dark` = dark), so all components read semantic tokens, never raw hex.
3. Add the Phase 1 component set (see "Component Inventory"): `button input label table badge dialog card dropdown-menu sonner (toast) form select switch skeleton avatar tooltip separator`.

### Token Architecture (primitive → semantic → component)

- **Primitive tokens** (raw scales): the neutral ramp, the brand teal ramp, status hues, the spacing scale, the type scale. Defined once; never used directly in components.
- **Semantic tokens** (intent): `--background`, `--foreground`, `--card`, `--muted`, `--border`, `--ring`, `--primary`, `--destructive`, `--success`, `--warning`, `--info`. These map to primitives per theme. **Components read ONLY semantic tokens.**
- **Component tokens** (local overrides): e.g. status-pill background/foreground per status, sidebar active-item background. Derived from semantic tokens.

This three-tier structure is the contract every later phase extends — add new semantic tokens (e.g. `--chart-1..6` for Phase 4 dashboards) rather than hardcoding values.

## Spacing Scale

8-point scale (all multiples of 4). Maps to Tailwind defaults so executor uses standard utilities.

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `1` | Icon-to-text gaps, badge inner padding, inline chip spacing |
| sm | 8px | `2` | Compact stacks, table cell vertical padding, form-field internal gaps |
| md | 16px | `4` | Default element spacing, card padding (mobile), input height rhythm |
| lg | 24px | `6` | Card padding (desktop), section padding, form-group spacing |
| xl | 32px | `8` | Layout gaps, gap between page header and content |
| 2xl | 48px | `12` | Major section breaks, login card vertical centering offset |
| 3xl | 64px | `16` | Page-level top/bottom breathing room, empty-state vertical padding |

**Layout constants (global shell — reused every phase):**
- Sidebar width: 240px expanded / 56px collapsed (icon-rail). Top bar height: 56px.
- Content max-width: 1280px, centered, with 24px (`lg`) gutters; tables may go full-bleed within the content column.
- Default border-radius: 8px (`rounded-lg`) for cards/inputs/buttons; 6px for badges/pills; 9999px for avatars and status dots.

**Exceptions:**
- Interactive icon-only targets (collapse toggle, row action menus, user-menu trigger) get a **44px** minimum hit area via padding even when the glyph is 16–20px (accessibility — not a new spacing token, a min hit-area rule).
- 1px hairline borders/dividers are not spacing and are exempt from the multiple-of-4 rule.

## Typography

Geist Sans for UI text; Geist Mono for technical/numeric values. Four roles only; two weights only (400 Regular, 600 Semibold). Tabular numerals (`font-variant-numeric: tabular-nums`) on all tables and numeric fields so digits align in dense ops views.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 | 1.5 (21px) | Default UI text, table cells, form values, descriptions |
| Label | 12px | 600 | 1.4 (17px) | Field labels, table column headers, badge text, nav section captions (uppercase, 0.04em tracking) |
| Heading | 20px | 600 | 1.3 (26px) | Page titles, dialog titles, card headers |
| Display | 28px | 600 | 1.2 (34px) | Login brand moment, top-level page hero (sparing) |

- **Mono override** (same sizes, `Geist Mono`, weight 400): IDs, hex codes, camera az/el/focal/fstop, sample counts, job/request tokens, env-var names in error copy.
- 14px body (not 16px) is deliberate: this is an information-dense internal ops console (Vercel/RunPod register), not a consumer marketing surface. 14px at 1.5 line-height meets readability for the target professional user.
- Min interactive text size 12px (labels only); never smaller.

## Color

**60 / 30 / 10 discipline.** Neutral surfaces dominate (Vercel/Notion calm); the teal accent is RARE and reserved for true brand/primary actions and active state. Accent **is teal — explicitly NOT purple/indigo.** Justification: a precise blue-green **teal/emerald** reads as technical, calm, and "instrument panel," fits a jewelry-render-ops tool (cool, precise, not flashy), is distinct from the generic AI-purple look, and leaves warm hues free to act as semantic status (amber/red) without clashing.

### Dark mode (default for the ops console — RunPod/Vercel feel)

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0A0A0A` | App background (`--background`) |
| Secondary (30%) | `#141414` cards / `#0F0F0F` sidebar+topbar / `#1F1F1F` borders | Cards (`--card`), sidebar, nav, table surfaces, `--border` |
| Accent (10%) | `#14B8A6` (teal-500) | See "Accent reserved for" |
| Destructive | `#F87171` (red-400) on `#2A1413` | Destructive actions only (disable user, cancel) |
| Foreground | `#FAFAFA` primary / `#A1A1AA` muted | Text (`--foreground`), secondary text (`--muted-foreground`) |
| Ring (focus) | `#14B8A6` at 2px, 2px offset | Keyboard focus — sharp, high-contrast (Vercel) |

### Light mode (also required — define both)

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` | App background |
| Secondary (30%) | `#FAFAFA` cards / `#F4F4F5` sidebar+topbar / `#E4E4E7` borders | Cards, sidebar, nav, table surfaces, borders |
| Accent (10%) | `#0D9488` (teal-600) | Same reserved uses; darker for AA on white |
| Destructive | `#DC2626` (red-600) | Destructive actions only |
| Foreground | `#18181B` primary / `#71717A` muted | Text, secondary text |
| Ring (focus) | `#0D9488` at 2px, 2px offset | Keyboard focus |

### Semantic status colors (shared by every phase — status pills, job/batch dashboards)

Defined now because the Job Monitor (Phase 4), Gallery (Phase 5), and batch progress need a stable status vocabulary. Phase 1 uses Admin/Operator role badges + active/disabled user state from this set.

| Status token | Dark fg / bg | Light fg / bg | Meaning |
|--------------|--------------|---------------|---------|
| `success` | `#34D399` / `#0E2A22` | `#15803D` / `#DCFCE7` | completed, active user |
| `warning` | `#FBBF24` / `#2A220E` | `#B45309` / `#FEF3C7` | queued, preview-quality notice, alpha-coverage warning |
| `info` / running | `#38BDF8` / `#0C2230` | `#0369A1` / `#E0F2FE` | running, in-progress |
| `destructive` / failed | `#F87171` / `#2A1413` | `#DC2626` / `#FEE2E2` | failed, cancelled, disabled user |
| `neutral` / idle | `#A1A1AA` / `#1F1F1F` | `#71717A` / `#F4F4F5` | draft, not-started, Operator role |
| `accent` / Admin | `#2DD4BF` / `#0E2A28` | `#0D9488` / `#CCFBF1` | Admin role badge, primary emphasis |

**Chart palette (declared now for later data viz; not rendered in Phase 1):** `--chart-1..6` = teal `#14B8A6`, cyan `#38BDF8`, amber `#FBBF24`, rose `#F87171`, emerald `#34D399`, slate `#94A3B8`. Categorical, color-blind-considerate ordering (teal/amber/rose are distinguishable). No purple in the ramp.

**Accent reserved for** (explicit — never "all interactive elements"):
- The primary button (one per view max: "Sign in", "Create user", "Save changes").
- The active/selected sidebar nav item (left indicator bar + tinted background + teal icon).
- Keyboard focus rings.
- The brand mark / wordmark in the top bar and login.
- The Admin role badge and "active" success states use their own semantic status colors, NOT the teal accent — accent stays scarce.

**Contrast:** all text/background pairs above meet WCAG AA (≥4.5:1 for body, ≥3:1 for ≥20px/large and for UI component boundaries). Teal-600 on white and teal-500 on near-black both clear AA for non-text UI and large text; body text never uses teal on a teal-tinted surface.

### Elevation & Motion (global, reused every phase)

- **Elevation** (dark mode leans on borders + subtle lift, not heavy shadows):
  - `flat`: border `1px var(--border)`, no shadow (cards, table containers).
  - `raised`: `0 1px 2px rgba(0,0,0,0.3)` + border (dropdowns, popovers).
  - `overlay`: `0 8px 24px rgba(0,0,0,0.45)` + border + backdrop blur (dialogs/modals, with a `rgba(0,0,0,0.6)` scrim).
- **Motion** (calm, fast, functional — Notion/Vercel restraint):
  - Durations: 120ms (hover/press feedback), 180ms (dropdown/menu), 220ms (dialog/sidebar collapse).
  - Easing: `cubic-bezier(0.2, 0, 0, 1)` (standard ease-out).
  - Honor `prefers-reduced-motion`: disable transforms/translate, keep opacity only.
  - No decorative/looping animation; spinners and skeletons only for genuine async waits.

## Phase 1 Surfaces

Concrete screens built on the system above. Each names its loading / empty / error / in-progress states (required by UI-02 even though UI-02 lands in Phase 7 — the foundation must not ship state-less surfaces).

### 1. App Shell (hosts every future phase)
- **Top bar (56px):** left = wordmark + workspace name ("Jewelry Render Studio"); right = theme toggle (dark default), user avatar menu (identity, role badge, Log out).
- **Left sidebar (240px / 56px collapsed):** nav items, each icon (lucide) + label, with a "MAIN" and "ADMIN" section caption (Label type, uppercase):
  - MAIN: **Products** (`box`), **Batches** (`layers`), **Jobs** (`activity`), **Gallery** (`image`)
  - ADMIN (Admin role only): **Domain Settings** (`sliders-horizontal`), **Users** (`users`)
  - Bottom-pinned: **Settings** (`settings`).
- Operator never sees the ADMIN section (and server still enforces — UI hiding is not the security boundary, per AUTH-05).
- Active item: left 2px teal indicator bar + `--accent`-tinted bg + teal icon.
- **States:** route-level loading = content-area skeleton (sidebar/topbar stay); error = inline error card with retry; the shell itself never blocks on a single panel.

### 2. Login Page (brand moment, internal-tool feel)
- Centered single card (max-width 400px) on the dominant background; wordmark + Display-size product name above; subtle, NOT consumer-marketing.
- Fields: **Email**, **Password** (show/hide toggle). One primary button: **"Sign in"**.
- **States:** default; field validation (inline, on blur/submit); **loading** (button shows spinner + "Signing in…", fields disabled); **error** (single calm banner above the form — see copy contract; never reveal whether email vs password was wrong); success → redirect to `/` (Products).
- Unauthenticated access to any route redirects here (with `?from=` preserved) — this is the visible side of SEC-03 deny-by-default.

### 3. User Management — Admin (`/admin/users`)
- Page header ("Users") + primary button **"Create user"** (opens dialog: email, temporary password, role select Admin/Operator).
- **Table:** columns Email · Role (badge) · Status (active/disabled pill) · Created · Actions (row `…` menu: Assign role, Disable/Enable user). Numeric/date columns mono + tabular.
- **States:** loading = 5 skeleton rows; empty = empty-state (see copy); error = inline error card + Retry; per-row action in-progress = row-level spinner, optimistic disable.
- **Disable user** and **role change to/from Admin** are destructive/sensitive → confirm dialog.

### 4. Domain Settings — Admin (`/admin/settings`)
Read + edit of the seeded domain (DATA-03). Tabbed or sectioned editable tables; a **sticky save bar** appears when dirty ("Save changes" primary + "Discard" ghost).
- **Camera views (4):** rows view1–4 with mono number fields az / el / focal(mm) / fstop. (Seed: v1 az30/el25/187.5/f2.8, v2 az180/el15/187.5, v3 az−30/el10/50, v4 az0/el75/187.5.)
- **Metals (3):** white / yellow / red with editable label + **color swatch** + mono hex.
- **Object groups (4):** alloycolour / diamond / stone2 / stone3 (read-only identifiers + editable display label).
- **Quality presets:** preview 64 / medium 256 / high 512 / ultra — editable sample count (mono number). Default resolution 1920×1920 shown.
- **States:** loading = skeleton table; error = inline + Retry; saving = save-bar button loading; success = success toast; validation = inline field errors (e.g. focal must be > 0).

### 5. Access / Role affordances
- **Operator vs Admin:** ADMIN nav section + admin pages are absent for Operators; an Operator who deep-links an admin route gets a calm 403 surface ("You don't have access to this area." + link back to Products) — mirroring the server-side 403 (AUTH-05), never a raw error.
- **Unauthenticated → login redirect** as in Login states above.
- **Logout** available from the user menu on every page (AUTH-02); clears session and returns to login.

## Copywriting Contract

Tone: plain, calm, operational, second-person, lowercase-friendly sentences (Notion/Vercel register). State the problem and the next action. No exclamation marks, no marketing voice, no blame ("Couldn't…" not "You failed to…").

| Element | Copy |
|---------|------|
| Primary CTA (login) | **Sign in** |
| Primary CTA (users) | **Create user** |
| Primary CTA (settings) | **Save changes** |
| Empty state heading (users) | **No users yet** |
| Empty state body (users) | Create the first account to give your team access. Only Admins can manage users. → [Create user] |
| Empty state heading (generic list, e.g. products/batches later) | **Nothing here yet** |
| Empty state body (generic) | When you {create your first product / build a batch}, it'll show up here. → [primary action] |
| Error state (login) | We couldn't sign you in. Check your email and password and try again. |
| Error state (data load) | Couldn't load {users / settings}. Check your connection and try again. → [Retry] |
| Error state (config / env, fail-fast) | The app is misconfigured: a required setting is missing. Contact an Admin. *(server logs the specific env var; UI never prints secret names to end users)* |
| Forbidden (Operator hits Admin route) | **You don't have access to this area.** This section is Admin-only. → [Back to Products] |
| Destructive confirmation — disable user | **Disable user:** "{email} will lose access immediately and can't sign in until re-enabled. Disable this account?" → [Cancel] [Disable user] (destructive) |
| Destructive confirmation — change to Admin | **Make Admin:** "{email} will be able to manage users and domain settings. Continue?" → [Cancel] [Make Admin] |
| Saving feedback | Toast: "Changes saved." / on fail: "Couldn't save changes. Try again." |
| Logout | Menu item: **Log out** |
