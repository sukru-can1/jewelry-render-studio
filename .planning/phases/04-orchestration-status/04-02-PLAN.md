<objective>
Implement the two DB-writing status paths: the 60s-safe chunked cron dispatcher (ORCH-01) and the idempotent webhook receiver (ORCH-02). These are the primary "submit" and "primary status" movers in the three-mover model; the reconcile/retry fallback ships in Wave 2.

Purpose: The Vercel 60s cap makes synchronous dispatch of a 48-job batch impossible (RESEARCH Pattern 1) — a bounded per-tick chunk is the only correct site. RunPod's webhook delivers terminal state push-style but at-least-once, so every write must be idempotent (RESEARCH Pitfall 3). The webhook URL is the primary status path, so the dispatcher must refuse to submit when it cannot build a valid one (A5) rather than silently killing status updates with a `https://undefined` callback.
Output: `lib/orchestration/dispatch.ts` + `app/api/cron/dispatch/route.ts`; `lib/orchestration/webhook.ts` + the adapted `app/api/webhooks/runpod/route.ts`. Turns test/orch-dispatch.test.ts and test/orch-webhook.test.ts GREEN.
</objective>
