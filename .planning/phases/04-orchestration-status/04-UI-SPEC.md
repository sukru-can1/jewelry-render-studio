## DB-only status model (executor-critical, not a redefinition)

The defining interaction constraint of this phase: **every page load and refresh in the monitor reads Postgres ONLY — never RunPod directly.** Status is written into the DB out-of-band by (a) the secret-verified RunPod **webhook** (the one public M2M route, SEC-04/ORCH-02) and (b) the **Vercel Cron reconciliation** fallback. The UI surfaces that DB state and communicates *freshness*, never liveness via per-request RunPod fan-out (ORCH-02 forbids it; the Vercel 60s cap makes it impossible at batch scale).

| Concept | What the UI does | What the UI must NEVER do |
|---------|------------------|---------------------------|
| Job/batch status | Read `Job.status` / derived batch status from the DB on load + on refresh | Call RunPod `/status/:id` per row on page load |
| "Live-ish" updates | Client **polls the DB** (lightweight `GET` that re-reads Postgres) on an interval, OR Next.js revalidation; surfaces an **"updated Ns ago"** freshness chip + manual **Refresh** | Open a socket to RunPod; fan out N RunPod calls |
| Terminal jobs (`completed`/`failed`/`cancelled`) | Treated as settled — excluded from refresh churn; never re-fetched | Re-poll a terminal job |
| Retry state | Render `Job.attempt` (e.g. "attempt 2 of 2") read straight from the DB | Trigger retries from the client; retries are server/cron-driven (ORCH-03) |

**Job-status → inherited status-pill mapping (DO NOT invent colors).** The DB enum (`prisma/schema.prisma`: `queued | submitted | in_queue | in_progress | completed | failed | cancelled`) collapses to the operator-facing 5-state vocabulary using Phase 1 status tokens:

| DB `JobStatus` value(s) | Operator-facing state | Inherited Phase 1 status token | Pill dot |
|--------------------------|-----------------------|--------------------------------|----------|
| `queued`, `submitted`, `in_queue` | **queued** | `warning` (amber) | amber |
| `in_progress` | **running** | `info` (cyan/blue) | blue, subtle pulse |
| `completed` | **completed** | `success` (green) | green |
| `failed` | **failed** | `destructive` (red) | red |
| `cancelled` | **cancelled** | `neutral` / idle (greyed) | grey |

**Derived batch status (computed from its jobs, not stored as a 6th color):** `queued` (all non-terminal, none running) · `running` (≥1 running, not all terminal) · `completed` (all completed) · `partly failed` (all terminal, ≥1 failed, ≥1 completed → `warning`) · `failed` (all terminal, all failed → `destructive`) · `cancelling`/`cancelled` (cancel requested/applied → `neutral`). The batch pill reuses the same inherited tokens; "partly failed" maps to `warning`, never a new hue.

## Color

**Inherited unchanged from Phase 1/2/3** — dark default + light, teal accent (NOT purple), 60/30/10 discipline, the full semantic + status vocabulary. P4 introduces **no new hues**; the whole point of the phase is to render the **already-declared status vocabulary**. The aggregate progress bar is the one place multiple status colors appear together, and it uses the inherited `success`/`warning`/`info`/`neutral` tokens as stacked segments — not new colors, and ordered color-blind-safely per Phase 1.

| Role | Value (dark / light) | Phase-4 usage |
|------|----------------------|---------------|
| Dominant (60%) | `#0A0A0A` / `#FFFFFF` | monitor + batches-list background |
| Secondary (30%) | `#141414` card · `#1F1F1F` border / `#FAFAFA` card · `#E4E4E7` border | batch-list rows/cards, aggregate panel surface, jobs-table surface, error-log viewer surface (slightly inset) |
| Accent (10%) | `#14B8A6` / `#0D9488` | see "Accent reserved for" |

### Aggregate progress bar — segment colors (inherited status tokens only)

The bar is a single horizontal track segmented by job outcome, segments ordered **completed → running → queued → failed** so green growth reads left-to-right and red failures pool at the right edge:

| Segment | Inherited token | Meaning |
|---------|-----------------|---------|
| completed | `success` (green) | jobs done |
| running | `info` (blue) | in progress (subtle indeterminate shimmer on this segment only, reduced-motion → static) |
| queued | `warning` (amber) | not started yet |
| failed | `destructive` (red) | failed (after retries exhausted) |
| cancelled | `neutral` (grey) | cancelled — shown as a hatched/greyed remainder, not a "loss" |

The numeric truth lives in the stat row beneath; the bar is the at-a-glance proportion. A 100%-complete bar is fully green; an all-failed bar is fully red; the common "mostly done, a couple failed" reads as a long green bar with a small red tail — calm, not alarming.

**Accent reserved for (P4 — extends Phase 1/2/3's list, does not broaden it):**
- The active sidebar **Batches** / **Jobs** nav item (inherited shell behavior).
- Keyboard focus rings (inherited 2px teal).
- The **manual Refresh** affordance and the freshness chip's "live" tick are **NOT** teal — they use `--muted-foreground`; refresh is a quiet utility, not a primary action.
- The **"Retry failed jobs"** button (if in scope, see surfaces) is a **secondary** button (neutral surface + border), NOT teal and NOT destructive — it is a corrective, non-primary action. The one place a primary teal button could appear on these surfaces is a forward CTA like **"View in gallery"** once a batch completes (reserved P5 link); even that is optional and may be a secondary link.

**Accent stays scarce:** the monitor is intentionally accent-light. Status meaning is carried entirely by the inherited semantic status colors (green/blue/amber/red/grey), never by teal. Teal would dilute the "instrument panel" reading where green=good and red=needs-attention must dominate. Failures are red but **calm** — non-blinking, paired with a clear next action, never a full-bleed alarm.

## Phase 4 Surfaces

Each surface names its loading / empty / all-queued / in-progress / mixed / all-done / all-failed / cancelling states (state-completeness is the inherited bar; this phase is *about* states).

### 1. Batches list — `/batches` (the inherited **Batches** nav item, currently a placeholder)

- **Header:** Heading **"Batches"** + a mono count on the left. (Optional, lightweight) a status filter / product filter — not required for MVP; the list defaults to the current operator's batches, most-recent first. No primary "create" button here — batches are created from a product (Phase 3); an optional secondary "Build a batch" link may point back to Products.
- **Body:** a list (table or stacked cards) of batches, each row showing: **product name** (Body) + small thumbnail (40×40, reused), the **combo summary** (mono, e.g. `3 metals × 4 angles × 3 passes = 36 jobs` — read back from `Batch.matrix` / `jobCount`), the **compact aggregate progress bar + `12 / 36`** mini variant, the **batch status pill** (derived, inherited tokens), **created** time (mono, relative), and the whole row is the **reopen** affordance → `/batches/[id]`.
- **States:** **loading** = 6 skeleton rows (inherited Skeleton); **empty** = centered empty-state ("No batches yet" + body pointing to building one from a product → [Go to products]); **error** = inline error card + Retry (inherited data-load copy); per-row the mini bar + pill communicate progress at a glance (all-queued amber / in-progress mixed / done green / failed red / cancelled grey). This list reads the DB only; it does not poll RunPod.

### 2. Batch detail / jobs monitor — `/batches/[id]` (THE centerpiece; the reserved Phase 3 link target)

- **Header:** Heading = **product name** + a mono batch id; the **derived batch status pill**; the **freshness indicator** ("updated Ns ago" + Refresh) on the right; the **Cancel batch** control (destructive secondary, enabled only while cancelable jobs exist); and — when the batch is terminal-with-failures — the optional **"Retry failed jobs"** secondary button. A context subline shows the combo summary (mono, from `Batch.matrix`) and quality.
- **Aggregate block:** the **aggregate progress bar + stat row** (segmented bar + completed / failed / running / queued / total). This is the operator's "how's it going" glance.
- **Jobs section:** the **status filter toggle group** (All / Queued / Running / Completed / Failed / Cancelled, with mono counts per tab) above the **jobs status table**. Each row = combo · status pill · attempt · duration · actions. Rows expand:
  - **Failed job → error-log viewer** (collapsible mono stdout/stderr tail + one-line plain summary + Copy).
  - **Completed job → light preview** (a single 160×160 thumbnail of the rendered pass + a **"View in gallery"** link — the link target is the P5 Gallery surface, reserved here; this phase does NOT build the layered browser or per-layer download).
  - **Queued/running job →** no expansion body beyond a "waiting…/rendering…" line + (if running) live duration; the **Cancel job** action lives in the row `…`.
- **Refresh model (DB-only):** while the batch is non-terminal, the client **polls the DB** at the configured interval (recommend ~5s) and/or relies on revalidation; the webhook + cron are what actually move the numbers (the UI just re-reads). The freshness chip shows recency; manual Refresh forces a re-read. When the batch reaches a terminal state, **auto-refresh stops** and the chip reads a final "updated …". The page must function with auto-refresh off (manual Refresh always available) for users who prefer it / reduced-motion contexts.
- **States:**
  - **loading** = skeleton aggregate bar + skeleton stat numbers + skeleton table rows.
  - **all-queued** = full amber bar, stat row `0 / 0 / 0 / N / N`, every job pill = queued, a calm "Queued — waiting for GPU workers to pick these up." note; Cancel batch enabled.
  - **in-progress** = mixed segmented bar with running shimmer, some jobs running (blue, live duration) / some queued; freshness ticking.
  - **mixed (some failed, still running)** = bar shows green + blue + amber + a red tail; failed rows expandable to logs; non-alarming.
  - **all-done** = full green bar, `N / 0 / 0 / 0 / N`, batch pill = completed, freshness frozen, optional "View in gallery" forward link.
  - **all-failed** = full red bar, batch pill = failed, each row expandable to its log, the **"Retry failed jobs"** secondary action (if in scope) prominent-but-calm; copy explains next steps without blame.
  - **partly-failed (terminal)** = green bar + red tail, batch pill = `warning` "partly failed", retry-failed available.
  - **cancelling** = affected pills → cancelling, bar segments shifting to grey, Cancel button → loading; settles to **cancelled** (grey) for the cancelled jobs (completed ones stay green).
  - **error (page-level DB read failed)** = inline error card + Retry (the DB read failed — distinct from a *job* failing); never surfaces a RunPod error to the operator as a page crash.

### 3. Cancel & retry — controls within surfaces 1–2 (ORCH-05 / ORCH-03)

- **Cancel batch:** header destructive-secondary → inherited destructive-confirm dialog → cancels all still-cancelable (`queued`/`running`) jobs; already-`completed` jobs are kept (their outputs aren't thrown away); pills + bar reflect `cancelling → cancelled`. Calm copy names exactly what stops and what's kept.
- **Cancel job:** row `…` action, enabled only for `queued`/`running` → confirm → that single job cancels. Terminal jobs expose no cancel.
- **Auto-retry surface (required):** the Attempt column + "retried" marker make the automatic ~2× idempotent retry **visible** (ORCH-03) — the operator sees `attempt 2 of 2` and understands a transient failure is being handled, without any action.
- **Manual "Retry failed jobs" (optional this phase):** batch-level secondary, only on terminal-with-failures, re-queues failed jobs idempotently. If descoped, the auto-retry surface alone satisfies the "see the retry state" requirement and this control moves to a later phase.
