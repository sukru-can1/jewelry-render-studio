<objective>
Implement the third mover: the reconcile cron that recovers webhook-missed status (ORCH-02 fallback), sweeps stranded jobs (submit-then-persist-crash), and drives the idempotent automatic retry of failed-under-cap jobs (ORCH-03). The re-queued jobs flow back through the Wave 1 dispatcher next tick.

Purpose: A missed/failed webhook would otherwise strand a job forever (RESEARCH Pattern 3); a job whose submit succeeded but whose runpodJobId persist crashed would sit non-terminal with no join key and never be polled (W-1) — the reconcile cron is the only place that can recover both. Auto-retry handles transient GPU failures without operator action while the terminal-state guard guarantees a completed render is never duplicated (RESEARCH Pattern 4).
Output: `lib/orchestration/reconcile.ts`, `lib/orchestration/retry.ts`, `app/api/cron/reconcile/route.ts`. Turns test/orch-reconcile.test.ts and test/orch-retry.test.ts GREEN.

    - reconcileNonTerminal(): (a) findMany jobs where runpodJobId not null AND status in (submitted, in_queue, in_progress), take ≤50; for each call getRunPodStatus and apply the SAME guarded mapping the webhook uses (mapRunPodStatus + status notIn TERMINAL guard). A job RunPod reports terminal becomes terminal in the DB; an unknown status no-ops. (b) STRANDED-JOB SWEEP (W-1): updateMany jobs where status in (submitted, in_queue, in_progress) AND runpodJobId IS NULL AND submittedAt < now − STRANDED_THRESHOLD (e.g. STRANDED_THRESHOLD_MS = 2 min) → set status 'queued' (clear submittedAt) so the dispatcher re-submits them next tick; this recovers a submit-succeeded-but-persist-crashed job that has no join key to poll. Returns { polled, releasedStranded } counts.
    - RETRY_CAP = Number(process.env.RETRY_CAP ?? 2).
    - retryFailedUnderCap(): updateMany where { status:'failed', attempt:{ lt: RETRY_CAP }, batch:{ status:{ notIn:['cancelled'] }} } set { status:'queued', attempt:{ increment:1 }, error:null, runpodJobId:null }. Returns requeued count.
    - A completed/cancelled job is untouched by retry (where filters to 'failed' only); a failed job at attempt≥CAP stays failed (no duplicate successful render — Pattern 4 idempotency guarantee).
  </behavior>
  <action>
Create `lib/orchestration/reconcile.ts` exporting `reconcileNonTerminal(): Promise<{ polled: number; releasedStranded: number }>` per behavior. To avoid drift, import and reuse the webhook's update path: call `applyWebhookCallback({ id: job.runpodJobId, status: s.status, output: s.output, error: s.error })` for each polled job (it already carries the terminal guard + mapping). THEN run the stranded-job sweep (W-1): a single guarded updateMany that re-queues non-terminal jobs whose runpodJobId IS NULL and whose submittedAt is older than STRANDED_THRESHOLD (module const, ~2 min) — these are jobs the dispatcher claimed and (apparently) submitted but never persisted a runpodJobId for, so they can never be polled; setting them back to 'queued' lets the next dispatch tick re-submit. The reset of runpodJobId to null means the next dispatch mints a fresh RunPod job (overwriting the join key — at most one in-flight RunPod job per Job row at a time).
