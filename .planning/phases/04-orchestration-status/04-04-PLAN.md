<objective>
Implement cancel (ORCH-05): a `cancelRunPod` helper on the existing RunPod client plus `cancelBatch` / `cancelJob` Server Actions guarded by session + IDOR, calling RunPod /cancel for in-flight jobs and recording the cancelled state immediately. The cancelled-batch where-guards added in Waves 1–2 already prevent re-dispatch.

Purpose: An operator must be able to stop a runaway or mistaken batch without losing already-completed renders (RESEARCH Pattern 7 / UI-SPEC cancel copy). This plan owns lib/runpod.ts exclusively so it runs parallel to the reconcile/retry plan in the same wave.
Output: extended `lib/runpod.ts` + `lib/orchestration/cancel.ts`. Turns test/orch-cancel.test.ts GREEN.

Cancel state model (W-4 decision — IMMEDIATE / option A, chosen for simplicity and matching the existing single-write design): on cancel, a cancelable job is set to status 'cancelled' RIGHT NOW in the same write, and `cancelRequestedAt` is recorded as the AUDIT TIMESTAMP of when the operator requested the cancel (not a separate pending phase). The UI 'cancelling' label is only a brief client-side optimistic state (useTransition) that resolves to 'cancelled' on the next DB read — there is NO reconcile-confirmed two-step cancel. 04-05's `deriveBatchStatus` and 04-06's monitor MUST treat cancelRequestedAt-with-remaining-non-terminal-jobs as the transient 'cancelling' display only, with the persisted truth being the 'cancelled' job statuses; this keeps the three plans consistent on model A.
</objective>
